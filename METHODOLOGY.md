# Methodology v6

This file is the shortest honest explanation of what the repository methodology
really is.

It is not:

- a magic ranking recipe
- a hidden scoring model
- a black-box crawler that replaces operator judgment
- a complete SEO agency playbook in one file

It is:

- a practical operating framework for SEO, GEO, and AI discoverability
- a self-hosted app plus scripts plus docs system
- a way to turn audits into evidence, priorities, fixes, and rechecks
- a bilingual workflow for global and Russian-speaking markets

## The operating model

The repository treats discoverability as three connected layers:

1. Rankings
   Search-engine visibility, crawlability, indexing, technical hygiene,
   semantics, authority, and local search execution.
2. AI visibility
   Whether assistants and answer engines can resolve the entity, retrieve
   factual content, and cite the site or brand correctly.
3. Conversion and trust
   Whether traffic or citations land on pages that are clear, trusted,
   commercially usable, and aligned with reality.

The repository is strongest when these three layers are handled together.

## What the methodology includes

### 1. Audit system

- technical SEO
- information architecture
- semantics and intent
- content, E-E-A-T, and entity clarity
- local and Yandex-specific discoverability
- GEO and answer-surface readiness
- factual consistency and drift control
- reporting, proof, and task export

### 2. Execution system

- score and finding generation
- impact / effort / confidence prioritization
- owner-ready task bundles
- graph and dependency explanation
- approval-first CMS and delivery flow
- repeat audit and comparison loops

### 3. AI-agent system

- task packs
- prompt packs
- command surface
- machine-readable contracts
- self-check paths before claiming “done”

## What the methodology does not fully include

The repository covers these areas, but not yet at “final agency bible” depth:

- enterprise-grade competitive intelligence at scale
- log-file SEO as a first-class runtime
- mature link acquisition operating system
- fully automated AI brand monitoring across all providers
- hosted-SaaS convenience with no operator setup

That is why the repository should be read as:

- a strong framework and operating platform
- not a claim that every SEO subdiscipline is already exhausted here

## How to read the checklists correctly

The `checklists/` folder is intentionally short.

Those files are:

- execution cards
- QA reminders
- operator handoff summaries

They are not the full methodology by themselves.

For depth, every serious workflow should combine:

- one deep playbook
- one checklist
- one prompt or task pack
- one script or app flow
- one proof or case reference
