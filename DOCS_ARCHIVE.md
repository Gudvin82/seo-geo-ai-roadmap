# Docs Archive Policy

The repository accumulated many release and versioned documents across `v3.x`,
`v4.x`, and `v5.x`.

That history is useful, but it should not be confused with the shortest path
for a new user.

## Current entrypoints

Start here first:

1. [README.md](./README.md)
2. [METHODOLOGY.md](./METHODOLOGY.md)
3. [SCORING_EXPLAINED.md](./SCORING_EXPLAINED.md)
4. [PUBLIC_PRODUCT_READINESS.md](./PUBLIC_PRODUCT_READINESS.md)
5. [REAL_CASES.md](./REAL_CASES.md)
6. [START_HERE_FOR_AI.md](./START_HERE_FOR_AI.md)
7. [AI_TASK_PACKS.md](./AI_TASK_PACKS.md)

## What counts as archive material

These remain useful, but are not the first stop for newcomers:

- `docs/en/*-vXYZ.md`
- `docs/ru/*-vXYZ.md`
- historical release summaries
- version-specific rollout notes
- intermediate upgrade-response documents

## Why keep archive materials at all

- they preserve product evolution
- they show honest response to criticism and failures
- they help operators understand why some boundaries exist
- they keep public claims auditable

## How to read them correctly

Use archive documents when you need:

- release chronology
- historical reasoning
- proof of a feature entering the repo
- comparison between old and current positioning

Do not use archive documents as the first explanation of the project.

## v6 rule

From `v6.0.0` onward:

- root entrypoints explain the current product
- archive files remain available for evidence and history
- README should route newcomers into current docs first, not historical slices
